# Replay and Golden Sets

Diese Datei uebertraegt Replay-/Golden-Set-Denken auf die geplanten
`crypto-rpc`-Generatoren und user-facing Artefakte. Ziel:
Generator-Aenderungen sollen nicht nur punktuell gruen sein, sondern
gegen stabile Projektzustaende erneut abspielbar bleiben.

Stand 2026-06-02 existiert noch kein Generator und keine Golden-Suite.
Diese Regeln werden verbindlich, sobald `crypto-rpc-gen`, Proto-
Artefakte, Stub-Generatoren oder Runtime-Source-Generatoren entstehen.

## Scope

Replay gilt fuer alle Pfade, die generierte oder user-facing Artefakte
erzeugen oder aktualisieren:

- Protobuf-IDL fuer `cryptorpc.pkcs11.v1`
- spaetere Protobuf-IDL fuer `cryptorpc.kms.v1`
- Go-, Java-, Kotlin-Zielpfad- und C#-Client-Stubs
- Go-, Java-, Kotlin-Zielpfad- und C#-Server-Stubs
- optional generierte Runtime-Source-Artefakte
- optionales hexagonales Runtime-Source-Scaffold
- Kompatibilitaetsmatrix fuer unterstuetzte/unsupported PKCS#11
  Funktionen, Mechanismen und Attribute
- Backend-Profile und Beispielkonfigurationen
- Traceability-Artefakte von UC-1 bis UC-7 zu `RPC-*`-IDs und Evidence

## Golden Case

Ein Golden Case beschreibt einen reproduzierbaren Generatorlauf:

```text
Name:
RPC / ADR refs:
Input project state:
Pinned sources:
  - OASIS PKCS#11 header version:
  - mapping file version:
  - surface profile:
  - generator version:
  - toolchain image/hash:
Command:
Expected action:
Expected changed files:
Expected generated bytes or structural shape:
Expected idempotency result:
Expected safety behaviour:
Evidence test:
```

Nicht jeder Golden Case braucht externe `testdata`-Dateien. Kleine,
fokussierte Cases duerfen im Test selbst als In-Memory-Fixture stehen.
Externe Goldens sind Pflicht, wenn ein ganzes user-facing Artefakt
stabil bleiben soll oder wenn ein Diff im Test schwer lesbar wird.

## Replay Rules

- Jeder neue Generator oder jedes neue user-facing Artefakt bekommt
  mindestens einen Golden Case fuer Fresh-State und Double-Run-NoOp.
- Jede Aenderung an OASIS-Header-Pins, Mapping-Regeln oder Surface
  Profiles braucht einen Golden Case, der die erwartete Output-Aenderung
  sichtbar macht.
- Jeder Stub-Generator braucht Evidence fuer Go, Java, Kotlin-Zielpfad
  und C# gemaess `RPC-MVP-003` und `RPC-NONGOAL-007`.
- Jeder Runtime-Source-Generator braucht Evidence, dass hexagonale
  Adaptergrenzen, Transportprofile, Config, Observability/Audit und
  Backend-Adapter nicht vermischt werden.
- JSON/YAML/Proto-artige Artefakte pruefen Struktur und, wo user-facing
  relevant, stabile Sortierung.
- Whole-file Golden Snapshots werden normalisiert nur dort eingesetzt,
  wo Strukturpruefung nicht reicht. Normalisierung darf keine
  semantischen Unterschiede verstecken.
- Golden Updates sind eine bewusste Produktentscheidung. Slice, Test
  oder Commit muss sagen, warum sich der erwartete Output geaendert hat.
- Goldens duerfen keine PINs, Klartexte, Schluesselmaterialien oder
  Sensitive Attributes enthalten.
- SoftHSM-Replay belegt API- und Mapping-Korrektheit, nicht
  Produktionsfaehigkeit.

## Verification Evidence

Wenn ein Slice Generatorverhalten aendert, enthaelt die
Verification-Evidence nach [`harness/verification.md`](verification.md)
zusaetzlich:

```text
Replay / Golden:
- Affected generators:
- Affected RPC / ADR refs:
- Golden cases added:
- Golden cases updated:
- Golden cases replayed:
- Intentional output changes:
- Pinned sources:
- Toolchain image/hash:
- Secret/sensitive-data check:
```

## When to Add External testdata

Lege externe Fixtures unter dem relevanten Paket an, zum Beispiel
`internal/.../testdata/golden/<case>/`, wenn:

- ein erwartetes Artefakt laenger als ein kurzer Inline-String ist,
- mehrere Dateien zusammen einen Zustand bilden,
- ein CLI-/Generator-Output fuer Nutzer sichtbar stabil bleiben muss,
- ein Review sonst nicht erkennen kann, ob die Aenderung absichtlich ist,
- mehrere Sprachen aus demselben Input verglichen werden muessen.

Halte externe Goldens klein. Ein Golden soll einen Vertrag pinnen, nicht
ein komplettes Demo-Projekt konservieren.

## Review Hooks

Reviewer sollen bei Generator-Diffs fragen:

- Gibt es einen Fresh-State Golden Case?
- Gibt es einen Double-Run-NoOp Golden Case?
- Gibt es Evidence fuer alle betroffenen Sprachen?
- Sind OASIS-Header, Mapping, Surface Profile und Toolchain gepinnt?
- Sind Sortierung und Format stabil?
- Wurde ein Golden geaendert, ohne dass Slice oder Commit die
  Output-Aenderung begruendet?
- Enthalten Goldens versehentlich Secrets oder sensitive Werte?
